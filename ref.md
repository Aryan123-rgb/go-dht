# Minimal Kademlia DHT in Go

A minimal, educational implementation of a Distributed Hash Table (DHT) based on the Kademlia architecture, written in Go. This project demonstrates decentralized network discovery and key-value storage routing using gRPC for inter-node communication.



## Project Layout

```text
/go-dht
│
├── proto/
│   └── dht.proto         # Protobuf definitions for the 4 Kademlia RPCs
│
├── Makefile              # Commands to compile the proto files (make proto)
├── client.go             # Handles outbound network operations and iterative routing
├── main.go               # Entry point, CLI UI, and node initialization logic
├── routing_table.go      # Manages the 160 XOR-distance buckets and node discovery
└── server.go             # gRPC server implementation handling incoming RPCs

Architecture & Implementation DetailsThis implementation relies on a unified 160-bit ID space. Both network nodes and the data keys they store are hashed into this exact same mathematical space using SHA-1.To determine where data should live, the network calculates the distance between a node and a piece of data using the bitwise XOR operator: $d(x, y) = x \oplus y$. Data is stored on the node whose ID is mathematically closest to the data's ID.Network DiscoveryPassive discovery is built into the protocol. Every RPC request (Ping, Store, FindNode, FindValue) includes the sender's identity. Upon receiving an RPC, the DHTServer extracts the sender's info and adds it to its local RoutingTable, allowing nodes to organically map the network just by communicating.Iterative Lookup ($\alpha=1$)When retrieving a value, if a queried node does not have the requested key, it returns the contact information of known nodes that are mathematically closer to the target. The Client takes these new nodes, adds them to a shortlist, sorts them by distance, and recursively queries the next closest unvisited node until the data is found or the network is exhausted.Key Structs and Methods1. routing_table.goManages the node's view of the network.NodeId: A 20-byte (160-bit) array representing the SHA-1 hash of a node or key.RoutingTable: Contains 160 buckets ([MaxBuckets][]*proto.Node).BucketIndex(id1, id2): Calculates the highest differing bit between two IDs to determine which bucket a node belongs in.AddNode(node): Safely adds a newly discovered node to the correct bucket based on XOR distance.2. server.goThe listener. Implements the DHTServer interface generated by gRPC.Ping: Liveness check. Adds sender to routing table.Store: Saves a key-value pair to the local map[string][]byte.FindNode: Returns the closest known nodes to a target ID from the routing table.FindValue: Looks up a key locally. Returns the value if found; otherwise, acts like FindNode and returns the closest known nodes to continue the search.3. client.goThe active worker. Handles all outbound requests and routing logic.Bootstrap(address): Dials a known node and executes a FindNode for its own ID to discover its neighbors and populate the routing table.findClosestNodes(targetId): Calculates the XOR distance between the target and all known nodes to find the optimal destination.StoreNetwork(key, value): Hashes the key, finds the mathematically closest node, and executes a remote Store RPC.FindValueNetwork(key): Implements the iterative Kademlia lookup algorithm. It queries nodes, processes their returned shortlists, and recursively follows the closest nodes until the value is retrieved.Demo InstructionsTo observe network discovery and iterative routing in action, open three separate terminal windows in the project root.1. Start the Genesis Node (Terminal 1)Starts the network. It knows nobody.Bashgo run . --port 5001
2. Bootstrap Node 2 (Terminal 2)Connects to Node 1 to join the network.Bashgo run . --port 5002 --bootstrap 127.0.0.1:5001
3. Bootstrap Node 3 (Terminal 3)Connects to Node 1 to join the network.Bashgo run . --port 5003 --bootstrap 127.0.0.1:5001
4. Store DataIn Terminal 3, instruct the node to store a key-value pair. The network will mathematically calculate the closest node to the hash of key1 and route the storage request there.Plaintext[Node 3] ➜ store key1 val1
5. Retrieve DataIn Terminal 2, instruct the node to retrieve the key. Node 2 will check its local storage, fail, and iteratively query the network to find the node holding val1.Plaintext[Node 2] ➜ get key1
Concepts Not Implemented (For Now)To keep this implementation minimal and focused on core routing mechanics, the following production DHT features are omitted:k-buckets with LRU Cache eviction: To constrain bucket sizes and make the lookup complexity strictly $\mathcal{O}(\log n)$.UDP protocol for transmission: Currently using a TCP protocol layer via gRPC to avoid packet loss, MTU limits, and network state debugging.TTL and Republishing: Missing periodic republishing of the keys across the network to handle node churn and data expiration.Node Departure Updates: No handling or background pinging regarding nodes leaving the network, failing, or going offline.
